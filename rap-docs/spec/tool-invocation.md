# Tool Invocation

The runtime invokes tools via HTTP POST to the tool's endpoint URL. The request is signed with AWS SigV4 (for Lambda Function URLs) or uses whatever auth the tool requires.

## Request format

```json
{
  "operation": "subscribe_github_events",
  "arguments": {
    "owner": "acme",
    "repo": "api",
    "event_type": "pull_request"
  },
  "id": "call_abc123",
  "call_id": null,
  "rap_receiver_url": "https://rap-receiver.lambda-url.us-east-1.on.aws/",
  "group_id": "thread_xyz",
  "user_id": "user_42"
}
```

| Field | Type | Description |
|---|---|---|
| `operation` | string | The tool name, matching the tool definition registered with the runtime |
| `arguments` | object | Tool arguments as specified by the tool's JSON Schema |
| `id` | string | Unique tool call ID generated by the LLM |
| `call_id` | string \| null | Optional secondary call ID (model-specific) |
| `rap_receiver_url` | string | URL where the tool should POST results |
| `group_id` | string | Thread ID for routing the result back to the correct conversation |
| `user_id` | string \| null | The end user's identity, if available |

## Expected behavior

The tool **must** return HTTP 200 immediately. The response body is ignored by the runtime — it exists only to confirm receipt. In Lambda, this is achieved with response streaming:

```javascript
export const handler = awslambda.streamifyResponse(async (event, responseStream) => {
  responseStream.write('OK');
  responseStream.end();
  // ... async processing continues after response is sent
});
```

The tool then processes the request and sends the result via the RAP receiver. There is no timeout enforced by the runtime — the tool can take as long as it needs.

## Tool definitions

Tools are registered with the runtime as JSON Schema definitions:

```json
{
  "type": "lambda",
  "name": "subscribe_github_events",
  "description": "Subscribe to GitHub webhook events on a repository.",
  "parameters": {
    "type": "object",
    "properties": {
      "owner": { "type": "string", "description": "Repository owner" },
      "repo": { "type": "string", "description": "Repository name" },
      "event_type": { "type": "string", "description": "GitHub event type to filter" }
    },
    "required": ["owner", "repo"]
  },
  "function_url": "https://tool.lambda-url.us-east-1.on.aws/"
}
```

Tool definitions can be loaded from environment variables, SSM Parameter Store, or DynamoDB. The reference implementation stores them in DynamoDB and injects them at deploy time via CDK.
