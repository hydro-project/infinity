# Tool Invocation

The runtime invokes a tool by POSTing a JSON payload to the tool's HTTP endpoint.

## Request

```http
POST https://tool.example.com/invoke
Content-Type: application/json

{
  "operation": "subscribe_github_events",
  "arguments": {
    "owner": "acme",
    "repo": "api",
    "event_type": "pull_request"
  },
  "id": "call_abc123",
  "call_id": null,
  "callback_url": "https://agent.example.com/callback",
  "group_id": "thread_xyz",
  "user_id": "user_42"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `operation` | string | yes | The tool name, matching the tool's registered definition |
| `arguments` | object | yes | Tool arguments as defined by the tool's JSON Schema |
| `id` | string | yes | Unique tool call ID, generated by the LLM |
| `call_id` | string \| null | no | Optional secondary call ID (model-specific) |
| `callback_url` | string | yes | URL where the tool must POST results |
| `group_id` | string | yes | Conversation thread ID, must be included in all callback payloads |
| `user_id` | string \| null | no | End user identity, for authorization or personalization |

## Response

The tool **must** return HTTP 200 immediately. The response body is not read by the runtime — it exists only to confirm receipt. The tool then processes the request asynchronously and sends results to `callback_url` using one of the callback message types.

There is no timeout enforced by the runtime. The tool can take as long as it needs.

## Authentication

The authentication mechanism between runtime and tool is implementation-specific. The reference implementation uses AWS SigV4 signing for Lambda Function URLs. Other implementations might use bearer tokens, mutual TLS, or no auth at all for internal services.
